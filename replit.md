# Multi-Agent AI Dev System

## Overview
The Multi-Agent AI Dev System is a lightweight, self-fixing AI coding agent designed for low-spec hardware. It operates on a Plan-Execute-Observe-Fix loop and incorporates a multi-API router for automatic failover and cost-aware routing. The project aims to provide an autonomous, goal-driven agent capable of handling complex development tasks through task chains and self-management features.

## System Architecture
The system is built around a core agent loop (Plan ŌåÆ Execute ŌåÆ Observe ŌåÆ Fix) and leverages a multi-API router for model flexibility.
The architecture includes:
- **web_app.py**: Flask web interface (port 5000) for user interaction.
- **main.py**: Command-line interface entry point.
- **agent.py**: Implements the core agent loop.
- **router.py**: Manages multiple LLM APIs (Gemini, Groq, OpenRouter, NVIDIA, Together, Ollama) with priority-based routing.
- **tools.py**: Provides agent capabilities like file operations, shell access, and browser automation.
- **browser.py**: Handles Playwright-based browser automation.
- **memory.py**: Manages persistent memory using JSON and SQLite, including execution history, skills, and learnings.
- **config.py**: Centralized configuration management.
- **orchestrator.py**: Implements a dual-layer thinking system with an outer loop (THINK ŌåÆ PLAN ŌåÆ EXECUTE ŌåÆ VERIFY ŌåÆ REFLECT ŌåÆ ADAPT) and multi-agent collaboration roles (Planner, Executor, Debugger, Reviewer, Critic, Memory).
- **vector_store.py**: Provides a vector memory layer using Chroma for semantic recall of tasks, solutions, errors, and reflections.
- **tool_decision.py**: A controlled tool decision layer that allows the orchestrator to pre-fetch read-only workspace context before execution.
- **task_chains.py**: Manages autonomous task chains, enabling the agent to tackle long-running goals by breaking them into smaller, manageable tasks with status tracking, priority, and replanning capabilities.
- **goal_engine.py** (Phase 19): Self-driven goal generator. Analyzes recurring failures from `memory.executions` and spawns `system_generated=1` chains via TaskChainRunner. Off by default; gated by enabled-toggle, cooldown, daily cap, confidence threshold, and authoritative loop-guard (system chains never trigger another generation). Wired into the chain-completion hook in web_app.run_session and exposed via /api/goals, /api/goals/events, /api/goals/toggle, /api/goals/run-now.
- **code_intel.py** (Phase 20.1): Read-only Code Intelligence Engine. AST-parses Python files, builds a project index of functions / classes / methods / imports, and resolves an intra-project dependency graph (handles absolute, relative, and submodule-style imports). Mtime-keyed JSON cache (`code_intel_cache.json`). Pure module + CLI (`python3 code_intel.py [--full|--file PATH|--symbol NAME]`); no web routes, no execution, no file edits. Foundation for future Phase 20 features (autonomous dev loop, sandboxed runner, in-editor AI).
- **code_learning.py** (Phase 20.2): Code Learning Memory ŌĆö persistent errorŌåÆfix knowledge base. Backed by the new `code_lessons` table in memory.db (raw CRUD lives in `Memory.insert_code_lesson` / `update_code_lesson_fix` / `list_code_lessons`, all silent-on-error). The `CodeLearning` class wraps a Memory instance and exposes `record_failure`, `record_fix`, `record_lesson`, `find_similar` (difflib-based with path/line-number/hex normalization and short-text false-positive guard), `format_past_fixes` (renders a "PAST FIXES" prompt block), and `lookup_for_prompt` (one-shot retrieve+format). No new dependencies, no UI changes, no execution wiring yet ŌĆö designed to be called from the future runner (20.4) and critic/reflection loop.
- **code_runner.py** (Phase 20.4): Sandboxed Code Runner. Pure-Python subprocess executor for untrusted Python source generated by the autonomous dev loop. Public API: `run_code(code, **opts) -> RunResult` and `run_script(path, **opts)`. Each call gets its own `tempfile.mkdtemp(prefix="coderun_")` sandbox at mode 0700, cleaned up unless `keep_sandbox=True`. The child runs as `python -E -s -B -u main.py [result_file] [extra_argv...]` (deliberately *not* `-I`, which would imply `-P` and drop the script's directory from `sys.path`, breaking staged `extra_files`). On Replit, `sys.executable` is a Go-wrapper that needs ~1 GB virtual memory to start; the runner caches `_REAL_PYTHON = <sys.exec_prefix>/bin/python3` at import time so `RLIMIT_AS` doesn't kill child startup. POSIX limits applied in a `preexec_fn` (each `setrlimit` independently try/except'd so a single failed limit doesn't crash the run): `RLIMIT_CPU` = caller value or `ceil(timeout)+1`, `RLIMIT_AS` = 256 MB default, `RLIMIT_FSIZE` = 5 MB default, `RLIMIT_CORE` = 0 always, optional `RLIMIT_NOFILE` / `RLIMIT_NPROC`. The first thing the preexec does is `os.setsid()` so the entire descendant tree can be killed via `os.killpg(pgid, SIGKILL)` ŌĆö closes the architect-flagged Phase 20.3 grandchild gap (verified by a test that forks a grandchild, stores its PID, and confirms it's dead after parent timeout). **Output is read by a `_stream_capped` helper using two daemon threads (one per pipe) that enforce `max_output_bytes` DURING read in 64 KB chunks and SIGKILL the process group the moment either stream exceeds the cap** ŌĆö replaces the original `communicate()` which buffered everything in parent RAM before truncation (architect-flagged DoS blocker; flooding child can no longer OOM the host). Env is scrubbed to `PATH`, `HOME`, `TMPDIR`, `PWD`, `PYTHON*`-safety flags, `LANG`, `LC_ALL` plus caller-supplied `env=` overrides; `_INHERITED_ENV_KEYS = (LANG, LC_ALL)` is the only allow-list inherited from the parent. Optional language-level import gate (`restrict_imports=True`) prepends a preamble that monkeypatches `builtins.__import__` to reject `DEFAULT_BLOCKED_MODULES` (subprocess, multiprocessing, ctypes, socket, ssl, urllib, http, ftplib, smtplib, telnetlib, shutil ŌĆö deliberately NOT `os` / `sys`); documented as defense-in-depth, bypassable via `importlib._bootstrap`. `extra_files={relname: content}` are staged into the sandbox with absolute-path and `..`-segment rejection plus a defense-in-depth `realpath` check that the resolved path stays inside the sandbox. **Result-file readback (`result_file_name="result.json"` channel for Phase 20.3 integration) opens with `O_NOFOLLOW` and rejects anything that isn't `stat.S_ISREG`** ŌĆö closes the architect-flagged symlink-exfiltration blocker (child planting `result.json -> /etc/shadow` no longer leaks); size cap (default 5 MB) is enforced via `fstat(fd).st_size` before `os.read` against the same fd (TOCTOU-safe). `RunResult` carries `status` Ōłł {success, error, timeout, killed, rejected}, `output`, `error`, `exit_code`, `duration_sec`, `sandbox_dir`, `truncated_stdout`/`truncated_stderr`, and an `extra` dict for the result-file channel. Argument validation uses a uniform `_rejected()` factory; rejected inputs include non-string code, `timeout` outside `(0, 60]`, non-positive caps, and unsafe `result_file_name` paths. Verified end-to-end against the Phase 20.3 harness (test 21: `generate_tests` + `_build_harness` round-tripped through `run_code` with `result_file_name="result.json"`; envelope `complete=True`, all rows present including the `ZeroDivisionError` smoke-test row). Honest limitations documented in module docstring: no chroot/seccomp/firejail available, same-uid file *read* outside sandbox not preventable without container tools, `RLIMIT_AS` subject to overcommit, import gate is best-effort. **Standalone ŌĆö does NOT modify code_testing.py and is NOT yet wired into the autonomous dev loop or any web route**, per the user constraint. 25/25 test groups pass after two architect rounds (round 1 found two HIGH blockers ŌĆö DoS via communicate buffering and symlink exfiltration via result-file readback ŌĆö both fixed and verified; round 2 = PASS).

- **code_testing.py** (Phase 20.3): Self-Testing System. AST-introspects a function's signature ŌĆö including `def` and `async def`, positional-only / positional / keyword-only parameters with defaults, and type annotations such as `int`, `str`, `list`, `Optional[X]`, `Union[X, None]`, and PEP 604 `X | None` ŌĆö and generates two flavours of test cases: (1) **smoke tests** that vary one parameter at a time across edge values (`0`, `-1`, `""`, `[]`, `None`, ŌĆ”) and pass iff no exception is raised, and (2) **failure tests** that pass a deliberately wrong-typed value and pass iff *any* exception is raised.
